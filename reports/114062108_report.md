# AI search Minichess Project: Advanced Search & Optimization Report
本報告專注於本專案中棋類 AI 引擎的核心搜尋演算法實作、高效剪枝策略、靜態搜尋、自定義移動排序優化，以及與助教所提供之測試引擎的實戰對弈分析。

## 1. Core Search Algorithms (核心搜尋演算法)
我實作了兩套進階搜尋架構，皆基於極大極小（Minimax）的 Negamax 對稱形式：

### 1.1 Alpha-Beta Pruning (Alpha-Beta 剪枝)
藉由傳遞 $\alpha$（我方最低保障分）與 $\beta$（敵方最大容忍得分）窗口，在子樹搜尋中主動剪去不可能發生的分支（Beta Cutoff）：
```cpp
if (alpha >= beta) {
    break; // 剪枝發生
}
```
- **換手處理**：更換玩家時，搜尋窗口翻轉為 $[-\beta, -\alpha]$；同玩家連步時，窗口維持 $[\alpha, \beta]$。

### 1.2 PVS (Principal Variation Search / 主要路徑搜尋)
PVS 建立在「第一步往往是最好的」假設上，對移動進行極端優化：
1. **主要路徑（PV）**：對第一個合法步進行常規全窗口 $[\alpha, \beta]$ 搜尋。
2. **非主要路徑（Non-PV）**：對後續棋步，先以極窄的「零窗口（Null Window）」 $[\alpha, \alpha+1]$（換手時為 [-$\alpha$-1, -$\alpha$]）進行探路。若對手能輕鬆回絕，則能以極快速度觸發剪枝。
3. **重新搜尋 (Re-search)**：若零窗口測試意外爆分（Fail-High，回傳值大於 $\alpha$），代表該步優於第一步，此時必須以寬窗口 [score, beta]（或換手對稱窗口）重新搜尋以取得精準分數。
## 2. Quiescence Search & Horizon Effect (靜態搜尋與地平線效應)
### 2.1 地平線效應 (Horizon Effect)
當搜尋達到預設的最大深度（`depth = 0`）時，若盤面正處於激烈的子力交換過程中，強行終止搜尋並調用評估函數，會產生嚴重的「短視」判斷。AI 會因為看不見下一手「被吃子」的發展，而做出致命的決策。

### 2.2 靜態搜尋的解決方案 (QS)
我實作了 `quiescence` 輔助搜尋：當深度耗盡時，不直接回傳靜態評估分，而是進入僅搜尋「捕獲步（Captures）」的特殊階段，直到盤面達到沒有吃子事件的「平靜狀態（Quiet）」再進行評估。
- **Standing Pat (最低保障)**：計算當前盤面的靜態評估分。因為玩家可主動選擇不進行任何吃子，故將此分數作為下限（若 `stand_pat >= beta` 則直接剪枝）。
- 深度安全網：在 `quiescence` 中加入 `ply > 10` 的安全中斷 (`PVS` 中為 `ply > 8`)，防止在大混戰殘局中發生無休止的吃子遞迴，徹底杜絕效能崩潰。

## 3. High-Performance Move Ordering (自定義移動排序)
搜尋演算法的剪枝效率高度取決於「展開分支的順序」。若最優棋步被排在最後面，Alpha-Beta 會退化成純 Minimax，PVS 則會因為頻繁重搜而導致效能災難。

我在 `State` 類別中實作了 `order_moves(...)` 函數，採用 **MVV-LVA (Most Valuable Victim - Least Valuable Attacker)** 規則進行動態排序：
```cpp
// 優先搜尋吃子步，且用價值最低的棋子去捕獲價值最高的敵方棋子
score = 100000 + (piece_vals[victim] * 10 - piece_vals[attacker]);
```
- **實作位置**：此優化被封裝在 `State` 成員中，供根節點、遞迴節點及 QS 節點共同調用。
- **成果**：排序使最優走法能更早被探查，提高剪枝發生率，並有效降低 PVS 搜尋中的重搜機率。

## 4. Active Time Control & Anti-Corruption (主動時間管理)
為因應 CLI 對弈中嚴格的時間限制與逾時判負規則，我實作了雙重防禦機制：
1. **零微秒響應中斷**：在遞迴與 QS 的內部 for 迴圈中，每次迭代首要檢查全域超時訊號 ctx.stop。一旦為真，立即退棧中斷。
2. **防禦性分數污染保護**：超時中斷時，搜尋會回傳不完整的垃圾分數 0。我在根節點搜尋中加入了防禦性阻斷：
   ```
   if (ctx.stop) {
        break; // 拋棄此步未完成的估分，避免污染先前已算好的 best_move 與 best_score
   }
    ```
這確保了引擎在時間到的一瞬間，能安全、穩定地回傳當前已完成搜尋之最高深度中的最優棋步，絕不掛機或下出亂步。

## 5. CLI Process-Kill Mechanism & Safety Margin (進程殺死機制與潛在改進提案)

### 5.1 關鍵發現：cli.py 的硬終止機制與主動中斷盲區

藉由分析 `cli.py` 原始碼，我發現了 `[DEBUG] No bestmove found` 導致超時判負的系統層面原因：

- **作業系統級強行終止**：當單步計時（例如 `--time 200`）結束時，`cli.py` 腳本會直接對 AI 引擎的進程調用作業系統級別的 `kill()`。

- **無法回傳的最佳步**：雖然我在搜尋遞迴中加入了 `ctx.stop` 的主動中斷檢查（如第 4 點所述），但在 CLI 環境下，進程通常還來不及在微秒級的進程切換延遲（Context Switch Latency）中完成退棧並將 bestmove 寫入 stdout，整個進程即已被作業系統強制殺死。

- **助教引擎的特性觀測**：在對局測試中，助教的 `minimax-weak`、`minimax-strong` 與 `boss-pvs` 從未發生過 `No bestmove found` 情況。這說明了對手引擎內部可能具備極其穩定的自主時間控制機制，能在外部硬終止發生前主動退出。

### 5.2 改善應對策略提案：安全邊際時間 (Proposed Safety Margin Strategy)

為徹底避免在 CLI 硬終止環境下被判負，我提出了在 C++ 引擎層引入「自主安全邊際計時器」的架構優化可能性：

- **自主解析與守護執行緒**：在 `ubgi.cpp` 中，解析來自裁判系統的時間限制參數。在啟動主搜尋執行緒的同時，在 C++ 端開啟一個獨立的計時守護執行緒。

- **安全邊際（Safety Margin）預留**：若單步限時為 $T$（如 200ms），自主計時執行緒在到達 $T - 30\text{ms}$ 到 $T - 50\text{ms}$ 的臨界點時，主動將全域的 `g_ctx.stop` 設為 `true`。

- **預期效果**：此提案將使 C++ 搜尋主執行緒在外部 Python 觸發 `kill()` 之前（預留 30~50 毫秒的極大餘裕），便已完成所有安全退棧並將當前最優步寫入標準輸出（stdout），從系統層面上將 CLI 的超時判負率降低至 0%。

助教提供了 minimax-weak、minimax-strong 與 boss-pvs 三大測試引擎。我透過 GUI（可視化分析）與 CLI（模擬助教雙敗對局測試：持雙方持黑白輪流對決兩盤）進行測試。

### 6.2 最終提交決策剖析

這項測試結果揭示了本引擎在沒有實作置換表（TT）下的最終提交難題：

1. **若追求「高上限（High Reward）」**：
PVS 理論上是擁有擊殺 Boss 最強戰術能力的引擎。其零窗口機制能在特定對局中提供傲視群雄的超深算力。但其代價是重搜懲罰帶來的極高超時（No bestmove）風險。

2. **若追求「低風險、高穩定度（Robustness）」**：
AlphaBeta 由於時間波動小、線性耗時，更能在 CLI 強制殺進程的規則下存活。雖然搜尋深度較淺，但其在 GUI 對戰 Boss 中所展現的多局獲勝紀錄，證明了在沒有 TT 的限制下，其總體戰術平穩度高於 PVS。

**結論**：在助教採用 **CLI 雙輪輪流對弈且逾時直接 Process Kill 判負** 的嚴苛規則下，本團隊基於不容許任何一次超時被殺進程的比賽策略，最終傾向提交「帶有主動時間防禦與 MVV-LVA 排序的 AlphaBeta」作為主要決賽 submission，以求取最穩健的積分表現。雖然可能無法打敗boss，但能較穩定地打敗minimax-weak和minimax-strong。
